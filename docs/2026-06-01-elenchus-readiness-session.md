# Elenchus readiness session

Prototype of the state file Elenchus produces (design answers #3 + #4). Generated by running
the council on Elenchus's own design. **Standing verdict: NOT-READY**, but the surface has
shrunk to a single category (C). The author hit an honest "I don't know" on C3 and entered

---

## Category A — The adjudication gate — RESOLVED

Author's resolution (sharper than the drafted one): **the judge detects contradictions; it
does NOT certify readiness.** Readiness is the author's own call (confidence / understanding).
The council's job is to make that self-declaration honest by exposing contradictions and
unverified facts.

- [x] **A1** Self-opposition dissolved: "survive without I-don't-know" is no longer the gate;
  the author's confidence is. The judge only flags contradiction/fact errors.
- [x] **A2** Bluff residual accepted and owned by the human: a perfectly self-consistent
  confident bluff can survive. The tool cannot and does not claim to certify true readiness.
- [x] **A3** Pass condition = author's own confidence + a contradiction-free build summary.
- [x] **A4** Summary consistency = mechanical contradiction-scan vs. logged answers (the same
  contradiction role), not a quality judgment.

**Scope locked:** Elenchus is a *develop-understanding + expose-contradiction* tool, not a
*certify-readiness* tool.

## Category B — Facts & feasibility grounding — RESOLVED

- [x] **B1** Seats verify factual claims via **web search** (general feasibility) + **Context7
  MCP** (current library/framework/API docs — the class of fact the macOS-only check needs).
  Author may challenge a correction → re-verify.

## Category C — Loop & state mechanics — RESOLVED

- [x] **C1** File = state/backlog; stress-test = always live on fresh answers. No conflict.
- [x] **C2** Resume protocol = **scan-on-invoke** (the skill Reads the state file from a known
  path, e.g. `docs/elenchus/<premise>.md`, only when convened). No token cost / pollution on
  unrelated sessions. A project-scoped `SessionStart` hook with `matcher: "resume"` is an
  optional enhancement *only* if a whole repo is one Elenchus project — not the default.
- [x] **C3** The markdown file IS the checkpoint (survives any context clear). Per round: skill
  **Writes** state *during* the round → user `/clear` or `/compact` → next engagement skill
  **Reads** it back. **Gotcha:** never rely on a `SessionEnd` hook to persist (~1.5s timeout);
  write during the round, before clearing. A returning "I don't know" keeps the question open
  (no reset/close); the loop terminates only when the author self-declares ready. "Default 2
  rounds" is per engagement; re-entry across study sessions is unlimited and cheap.
